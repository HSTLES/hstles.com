Architecture

The Edge (Gateways)

Public-facing gateways handle thousands of simultaneous connections, terminate TLS, validate requests, and translate HTTP to internal protocols. They hold no database credentials — can't leak what they don't have.

  • ✓ Handle thousands of simultaneous connections
  • ✓ Terminate TLS and validate requests
  • ✓ Translate HTTP to internal protocols
  • ✓ No database credentials — can't leak what they don't have
  • ✓ Optimised for memory and connection handling

Edge Gateway Architecture

  • ✓ Client → HTTPS → Gateway
  • ✓ Gateway → Protobuf → VL1 Mesh
  • ✓ Credentials: none
  • ✓ DB Access: none
  • ✓ Secrets: none

Protocol break = natural firewall

Architecture

The Core (Workers)

Private-network workers execute business logic, handle database queries, manage secrets and credentials securely, and process computationally expensive operations. Not publicly accessible — isolated by design.

  • ✓ Execute business logic and database queries
  • ✓ Handle computationally expensive operations
  • ✓ Manage secrets and credentials securely
  • ✓ Not publicly accessible — isolated by design
  • ✓ Optimised for CPU and throughput

Core Worker Architecture

  • ✓ VL1 Mesh → Worker → Database
  • ✓ Internet Access: none
  • ✓ DB Credentials: secure vault
  • ✓ Signing Keys: secure vault

Isolated by design · no public exposure

Security

Zero-Trust by Design

Defence in depth with privilege separation at every layer. Public-facing gateways don't possess database passwords or signing secrets. The protocol break between HTTP and internal mesh acts as a natural firewall.

  • ✓ Credential isolation — gateways hold no secrets
  • ✓ Protocol break — HTTP to Protobuf enforces schema validation
  • ✓ No lateral movement — each request validated independently
  • ✓ VL1 overlay with Noise Protocol encryption
  • ✓ High resistance to data exfiltration from the edge

Zero-Trust Security Model

  • ✓ TLS 1.3 — Client ↔ Gateway
  • ✓ Protobuf — Schema validation
  • ✓ VL1/Noise — Gateway ↔ Worker
  • ✓ AES-256 — Data at rest

Compromise gateway = no DB access
Compromise worker = no internet access

Dual-Mode Operation

Distributed by default, monolith when needed. In mesh mode, services are distributed across multiple nodes. In direct mode, gateways execute backend logic via direct function calls — zero network dependency for emergency continuity.

  • ✓ Mesh Mode — horizontal scaling, geographic distribution
  • ✓ Direct Mode — "break glass" disaster recovery
  • ✓ Same business logic runs in both modes
  • ✓ No code changes required for failover
  • ✓ Independent component updates

Data Flow

How a request flows

From user click to database and back.

Ingress & Translation

User request hits Gateway via HTTPS. Gateway validates the request and converts it to Protobuf, enforcing strict schema validation at the protocol boundary.

Discovery & Transport

Gateway queries the directory to find a healthy Worker and establishes a secure VL1 session. Encrypted mesh transport ensures no eavesdropping or tampering.

Execution & Response

Worker executes business logic, queries the database, and returns the result back through the mesh to the Gateway and on to the user.
